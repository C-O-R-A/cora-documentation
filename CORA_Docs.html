Get Started

CORA Documentation

CORA (Collaborative Robot Arm) is a modular cobot platform. Design your arm in the browser, generate matching CAD and ROS 2 software automatically, build it, and run it.

ℹ️
Alpha release

CORA is under active development. The joint manifest schema is stable; the configurator UI and Python SDK are in progress. APIs may change between minor versions.

What is CORA?

CORA consists of three integrated layers that take a user from an idea to a running physical arm:

🎛️
Configurator
Browser-based app to compose joints, set parameters, and preview the arm in 3D.
📤
Export Pipeline
Auto-generates STEP, URDF/XACRO, MoveIt SRDF, ros2_control YAML, and Python SDK config.
🌐
Simulation
Gazebo simulation mirrors your exact arm. Validate motion plans before touching hardware.
🐍
Python SDK
High-level SDK for joint control and motion planning without writing ROS 2 directly.

Design principles

CORA is built around a small number of explicit constraints:

PrincipleWhat it means in practice
Schema-firstThe joint manifest JSON schema is the contract between the configurator, the export pipeline, and the physical build. Lock it before implementing anything else.
Drop-in extensibilityAdding a new joint type requires only a folder: manifest, mesh files, and a CadQuery script. No platform code changes.
Zero manual integrationEvery exported software package is generated directly from the hardware configuration. The URDF, SRDF, and control YAML always match the physical arm.
REP-103 everywhereZ-up, metres, radians throughout — no unit conversions anywhere in the stack.

Get Started

Quickstart

From zero to a simulated running arm in under 30 minutes.

1
Clone the repository
Get the monorepo and install dependencies for frontend and backend.
bash
git clone https://github.com/your-org/cora.git
cd cora

# Frontend
cd frontend && npm install && cd ..

# Backend
cd backend && pip install -e ".[dev]" && cd ..
2
Start the dev server
Run the FastAPI backend and Vite dev server concurrently.
bash
# Terminal 1
cd backend && uvicorn app.main:app --reload

# Terminal 2
cd frontend && npm run dev
3
Configure your first arm
Open http://localhost:5173. Add joints using the sidebar panel. The 3D viewport updates live.
4
Export your package
Click Export. You'll receive a ZIP containing STEP files, a URDF, MoveIt SRDF, and a ROS 2 package ready to build.
5
Launch in simulation
Build and source the package, then launch Gazebo + RViz2.
bash
cd ~/ros2_ws
colcon build --packages-select cora_robot
source install/setup.bash
ros2 launch cora_robot sim.launch.py
✅
Expected result

Gazebo opens with your configured arm. RViz2 shows the robot model with MoveIt 2 loaded. You can send joint goals from the Motion Planning panel.

Installation

CORA requires Node.js 20+, Python 3.11+, and ROS 2 Jazzy. The configurator and backend can run without ROS 2 installed.

Prerequisites

DependencyVersionRequired for
Node.js≥ 20.xFrontend configurator
Python≥ 3.11Backend, SDK, CadQuery
ROS 2JazzyGenerated ROS 2 packages, simulation
GazeboHarmonicSimulation only
MoveIt 2Jazzy branchMotion planning

ROS 2 workspace setup

bash
# Source ROS 2 first, then build the overlay
source /opt/ros/jazzy/setup.bash
cd ~/ros2_ws
colcon build --symlink-install
source install/setup.bash
⚠️
ABI mismatch warning

If you see a segfault in gz_ros2_control, remove build/, install/, and log/ and rebuild. Always source the system ROS 2 install before sourcing the local overlay — never reverse the order.

Configurator

Joint Manifest Schema

Every joint in the CORA library is defined by a manifest.json. This file is the contract between the configurator, the export pipeline, and the physical build instructions.

Top-level fields

FieldTypeDescription
id requiredstringUnique joint type identifier, e.g. revolute_80mm. Used as the folder name.
display_name requiredstringHuman-readable label shown in the configurator.
type requiredenumrevolute | prismatic | fixed
axis required[x, y, z]Joint axis vector in joint-local frame. REP-103: metres.
limits requiredobjectPosition, velocity, and effort limits. See below.
params optionalobjectJoint-type-specific parameters (gear ratio, motor model, etc.).
connectors optionalobjectElectrical connector layout for the back shell.
mesh requiredobjectPaths to visual and collision mesh files.

Example manifest

json
{
  "id": "revolute_80mm",
  "display_name": "Revolute 80 mm",
  "type": "revolute",
  "axis": [0, 0, 1],
  "limits": {
    "lower": -3.14159,
    "upper": 3.14159,
    "velocity": 1.5,
    "effort": 10.0
  },
  "params": {
    "gear_ratio": 50,
    "motor_model": "T-Motor AK10-9",
    "housing_diameter_mm": 80
  },
  "connectors": {
    "power_in":  "XT60_panel",
    "power_out": "XT60_panel",
    "can_in":   "M12_A5_bulkhead",
    "can_out":  "M12_A5_bulkhead"
  },
  "mesh": {
    "visual":    "mesh.dae",
    "collision": "mesh_collision.stl"
  }
}

Joint Library

revolute_80mm

The primary revolute joint in the CORA library. 80 mm housing, 50:1 gearbox, T-Motor AK10-9 actuator. Designed for desktop-scale manipulation tasks.

Specifications

ParameterValue
Housing diameter80 mm
Joint typerevolute
Rotation range±180° (−π to +π rad)
Max velocity1.5 rad/s
Max effort10.0 N·m
Gear ratio50:1
MotorT-Motor AK10-9

Connector layout

Four connectors are located on the back shell. Reading top-to-bottom on the back face:

ascii
 ┌──────────────────────┐
 │  [XT60 IN ] [CAN IN ] │  ← power chain in / CAN in
 │  [XT60 OUT] [CAN OUT] │  ← power chain out / CAN out
 └──────────────────────┘

CAN connectors: Binder Series 763 or Phoenix Contact SACC-M12. Power connectors: Amass XT60PW-M/F panel mount.

Joint Library

Adding a Custom Joint

CORA is designed to be extended without modifying platform code. A new joint type is a self-contained folder.

Folder structure

tree
backend/joints/
└── your_joint_id/
    ├── manifest.json    # required
    ├── cad.py           # CadQuery script
    ├── mesh.dae         # visual mesh
    └── mesh_collision.stl  # collision mesh
1
Create the folder
Use your joint ID (snake_case) as the folder name under backend/joints/.
2
Write manifest.json
Follow the Joint Manifest Schema. All required fields must be present; validation runs on server startup.
3
Write cad.py
A CadQuery script that accepts the params dict from the manifest and returns a STEP file. The export pipeline calls build(params) → Path.
4
Add mesh files
Provide a visual mesh (.dae or .stl) and a simplified collision mesh (.stl). Reference them in the manifest's mesh field.
5
Restart the backend
The joint appears in the configurator automatically on next server startup. No code changes required.

3-DOF Desktop Arm

A minimal arm configuration for pick-and-place on a desktop workspace. Three revolute_80mm joints in shoulder–elbow–wrist arrangement.

Configuration file

json

  "description": "3-DOF desktop pick-and-place arm",
  "joints": [
    {
      "name": "shoulder_pan",
      "type": "revolute_80mm",
      "origin": { "xyz": [0, 0, 0.08], "rpy": [0, 0, 0] },
      "params": { "gear_ratio": 50 }
    },
    {
      "name": "elbow",
      "type": "revolute_80mm",
      "origin": { "xyz": [0, 0, 0.15], "rpy": [0, 1.5708, 0] },
      "params": { "gear_ratio": 50 }
    },
    {
      "name": "wrist",
      "type": "revolute_80mm",
      "origin": { "xyz": [0, 0, 0.15], "rpy": [0, 1.5708, 0] },
      "params": { "gear_ratio": 50 }
    }
  ]
}

bash
ros2 launch cora_robot sim.launch.py config:=desktop_3dof_v1
Software

Python SDK

The CORA Python SDK provides a high-level interface for joint control and motion sequencing without writing ROS 2 directly. It wraps the generated ros2_control interfaces.

Installation

bash
pip install cora-sdk  # when published; or install from source
pip install -e backend/  # from monorepo root

Basic usage

python
from cora import CoraArm

# Initialise from your exported config
arm = CoraArm("desktop_3dof_v1")
arm.connect()

# Move a single joint by name
arm.move_joint("shoulder_pan", position=1.57)  # radians

# Move to a named pose
arm.go_to_pose("home")

# Cartesian waypoints (requires MoveIt 2 running)
arm.move_cartesian([
    (0.1, 0.0, 0.3),  # x, y, z in metres
    (0.2, 0.1, 0.3),
])

arm.disconnect()

Context manager

python
with CoraArm("desktop_3dof_v1") as arm:
    arm.go_to_pose("home")
    arm.move_joint("elbow", position=-0.5)
ℹ️
ROS 2 node lifecycle

CoraArm manages its own rclpy node. connect() calls rclpy.init() if not already initialised. If you are embedding the SDK inside an existing ROS 2 node, pass init_ros=False to the constructor.

Build Guide

Build Overview

This guide covers the full physical build process: sourcing components, printing parts, assembling joints, wiring connectors, and powering on for the first time.

Build sequence

1
Export your package
Use the configurator to finalise your arm and export. You'll need the STEP files for fabrication and the BOM for ordering.
2
Order components
Use the generated BOM. Motors and gearboxes have 2–3 week lead times; order early.
3
Print structural parts
STEP files are designed for FDM printing in PETG or ABS. Minimum 40% infill recommended for joint housings.
4
Assemble joints
See Mechanical Assembly. Each joint is self-contained and clips into the next.
5
Wire connectors
XT60 for power chain, M12 for CAN bus. See Electrical & Connectors.
6
First power-on
Follow the First Power-On checklist. Never power on without completing the pre-flight checks.
Build Guide

Electrical & Connectors

Each joint has four connectors on its back shell: two for the 60 A power chain (XT60, in + out) and two for CAN bus (M12 A-coded 5-pin, in + out).

Connector selection rationale

No single M12 connector family handles both 60 A power and CAN bus simultaneously. The two-connector-per-side solution is the practical outcome of this constraint:

FunctionConnectorRatingSourcing
Power (60 A)XT60 panel-mount60 A continuousAmass XT60PW-M/F, RAMPOW
CAN busM12 A-coded 5-pin bulkhead4-pin CAN + shieldBinder 763, Phoenix SACC-M12
⚠️
PCB clearance constraint

Axial PCB depth is approximately 12–18 mm on the revolute_80mm joint. All connectors must be sourced and verified against this clearance before ordering the PCB run.

Software

Gazebo Simulation

The simulation environment mirrors your physical arm exactly. Run motion plans in Gazebo before touching hardware to catch configuration errors early.

Launching

bash
source /opt/ros/jazzy/setup.bash
source ~/ros2_ws/install/setup.bash
ros2 launch cora_robot sim.launch.py

Sim clock

To suppress the sim-clock warning in RViz2, set use_sim_time: true for the RViz2 node in your launch file:

python
rviz_node = Node(
    package='rviz2',
    executable='rviz2',
    parameters=[{'use_sim_time': True}],
)

Motion planning with CHOMP

CHOMP planning is enabled by default in the MoveIt 2 configuration. To switch to OMPL, edit config/moveit/planning_pipeline.yaml and set planning_plugin: ompl_interface/OMPLPlanner.

Troubleshooting

Common issues and how to fix them.

Segfault in gz_ros2_control

⚠️
ABI mismatch — full clean rebuild required

This segfault occurs deep in STL destructor chains when locally built packages were compiled against a different ABI than the system ROS 2 install. Fix: remove build/, install/, and log/, source the system install first, then rebuild.

bash
cd ~/ros2_ws
rm -rf build/ install/ log/
source /opt/ros/jazzy/setup.bash
colcon build --symlink-install
source install/setup.bash

RViz2 sim-clock warning

Add parameters=[{'use_sim_time': True}] to the RViz2 node declaration in your launch file. See Simulation.

Onshape API rate limits

The free Onshape API tier enforces annual rate-limit caps. For production use, publish CORA as an Onshape App Store application using OAuth2 — this removes the annual cap. The EphemeralExporter in backend/app/onshape/ is designed to work with both auth paths.
